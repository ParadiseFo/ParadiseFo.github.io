# PRD：技术向个人博客 — 后端与管理后台（自建）

- **文档版本**：v0.1  
- **日期**：2026-04-11  
- **关联文档**：`docs/PRD.md`（Astro 前台 / SSG，**v0.2 起文章数据以后端为主**）  
- **产品形态**：自建 HTTP API + 关系型数据库 + 浏览器管理后台；面向「高访问量技术博客」常见扩展路径设计  
- **已定约束**：**自建**管理后台与数据存储；**在浏览器中发布/更新文章**；读者侧延续「多用户注册只读、无评论」的产品边界（与此前讨论一致）

---

## 1. 背景与目标

### 1.1 背景

前台（`PRD.md`）以静态或近静态方式交付内容，利于 CDN 与 SEO。若文章需 **在浏览器内编辑并发布**，则需要 **持久化存储与鉴权**，并配套 **管理后台**。访问量较大的技术向个人站点，其后端常见共性包括：**读写分离路径、缓存、限流、可观测性、备份**，本 PRD 将上述共性落实为可验收条款（实现阶段可选用具体技术栈）。

### 1.2 产品目标

- **发布体验**：授权用户在浏览器中完成 **草稿 / 发布 / 下线**（或等价状态），无需本地 Git 操作。  
- **读者体验**：公开文章列表与详情 **低延迟、高可用**；高峰时后端可 **水平扩展或有界降级**。  
- **安全与合规**：身份鉴别、权限隔离、常见 Web 与数据安全基线。  
- **可运维**：健康检查、日志、指标、备份与恢复流程可执行。

### 1.3 成功指标（上线后可量化）

| 维度 | 目标（首版建议） |
|------|------------------|
| 可用性 | 月度可用性目标由运维策略定义（例如 99.5%+）；计划内维护窗口单独公告 |
| 读接口延迟 | 公网 **p95** 在典型负载下 **< 300ms**（不含跨国极端网络；以监控为准） |
| 写接口延迟 | 发布/保存 **p95** **< 1s**（不含大附件上传） |
| 错误率 | 5xx 比例 **< 0.1%**（排除部署窗口） |
| 数据 | 每日自动备份 + 恢复演练 **每季度至少一次**（记录结果） |

> 注：个人博客流量差异极大，指标以 **可监控、可对比** 为主；上线后按真实数据调参。

## 2. 业界常见架构要点（调研摘要）

以下内容来自内容站 / API 扩展的通用实践，用于约束本 PRD，**不绑定**具体云厂商：

1. **边缘与静态**：HTML/CSS/JS 与媒体走 **CDN**；API 与动态页 **不缓存私人数据**。  
2. **读多写少**：文章公开读占绝大多数；采用 **列表/详情缓存**、**HTTP 缓存头**、必要时 **应用层缓存（如 Redis）** 降低数据库压力。  
3. **缓存失效**：发布/更新后 **主动失效**相关缓存（键失效或版本号）；避免「长期脏读」。  
4. **连接与查询**：数据库 **连接池**；列表分页；详情按 **slug/id 索引**；避免 N+1。  
5. **峰值保护**：**限流**（按 IP / 用户 / 路由）、**超时**、**熔断或降级**（只读降级到缓存或静态页策略）。  
6. **安全**：HTTPS、**HttpOnly Cookie** 或短期 **Bearer**、CSRF 防护（若 Cookie 会话）、**XSS**（富文本/Markdown 渲染策略）、**SQL 注入**防护（参数化查询/ORM）。  
7. **可观测性**：结构化日志、请求 ID、错误追踪；RED/USE 类指标（请求数、错误率、延迟；资源使用）。

## 3. 范围

### 3.1 In Scope（后端首版）

- **账户与角色**  
  - 注册/登录（读者）；**仅管理员/作者**可进入管理后台并发文。  
  - 密码存储使用 **强哈希**（如 Argon2/bcrypt，具体实现阶段选定）。  
- **文章生命周期**  
  - 字段至少：`title`、`slug`（唯一）、`summary`、`body`（Markdown 或受控富文本）、`status`（`draft`/`published`/`archived` 或等价）、`published_at`、`updated_at`、`author_id`。  
  - **浏览器内保存**草稿与正式发布；发布后读者可见。  
- **公开读 API（或 BFF）**  
  - 列表（分页、按时间倒序）、详情（按 slug 或 id）。  
  - **仅返回已发布**文章给匿名与普通读者。  
- **管理 API**  
  - 文章的增删改、状态流转；**仅特权角色**可调用。  
- **基础安全能力**  
  - 鉴权、授权中间件；管理接口 **更强限流**；审计日志（谁在何时发布了哪篇文章，至少记录管理员操作）。  
- **运维基线**  
  - `/health`（或等价）健康检查；日志可检索；数据库备份任务。

### 3.2 Out of Scope（首版不做）

- 评论、点赞、通知、站内信（与此前「无评论」一致，可后续单独立项）。  
- 全文搜索引擎集群（可用 **数据库 `LIKE`/简单索引** 或后续接 **Meilisearch/Elastic**）。  
- 多区域主主数据库（单区域 + 备份即可，除非你有合规要求）。  
- 支付、商业化。

### 3.3 与前台 `PRD.md` 的衔接方式（需实现前选定）

至少选一种并写入技术方案：

| 模式 | 说明 |
|------|------|
| **A. 发布后重建静态站** | 文章存数据库；发布 Webhook 触发 CI 执行 `astro build`，前台仍为纯静态。SEO 友好，**发布有分钟级延迟**。 |
| **B. 前台运行时拉取 API** | Astro/前端在构建或请求时从 API 取文章，SEO 需 **SSR/混合渲染** 或预渲染策略，工程复杂度更高。 |
| **C. 混合** | 首页列表静态 + 详情动态；或 ISR 类策略。 |

> 本 PRD 要求：**必须明确选型**，并在验收中体现（例如 A 模式需演示一次发布到前台可见的端到端流程）。

## 4. 用户与权限

| 角色 | 权限 |
|------|------|
| 访客 | 读公开已发布文章（若产品允许未登录阅读） |
| 读者（注册用户） | 同上；**不可**访问管理接口与草稿 |
| 管理员/作者 | 管理后台 + 文章写权限；可配置 **是否允许多作者** |

**禁止**：读者通过篡改请求访问 `draft` 或他人草稿。

---

## 5. 数据模型（逻辑层，实现可微调）

**users**：`id`、`email`（唯一）、`password_hash`、`role`（`reader`/`admin`/`author`）、`created_at`、可选 `email_verified_at`。

**posts**：`id`、`author_id`、`slug`（唯一）、`title`、`summary`、`body`、`status`、`published_at`、`created_at`、`updated_at`。

**sessions** 或等价鉴权存储：若用有状态 Session，需 **安全 Cookie** 与 **CSRF** 策略；若用 JWT，需 **短期 + 刷新** 与 **吊销** 策略说明。

**audit_logs**（推荐）：`actor_id`、`action`、`target_type`、`target_id`、`metadata`、`created_at`。

索引建议：`posts(status, published_at desc)`、`posts(slug)`、`users(email)`。

---

## 6. API 概要（示例，实现期可 OpenAPI 化）

**公开**

- `GET /api/v1/posts?page=&limit=` → 已发布列表  
- `GET /api/v1/posts/:slug` → 详情  

**认证**

- `POST /api/v1/auth/register`（若开放注册）  
- `POST /api/v1/auth/login`  
- `POST /api/v1/auth/logout`  

**管理（需特权）**

- `POST /api/v1/admin/posts`  
- `PATCH /api/v1/admin/posts/:id`  
- `POST /api/v1/admin/posts/:id/publish`（若拆状态机）  
- `DELETE /api/v1/admin/posts/:id`（或软删除，产品需定义）

**错误**：统一 JSON 结构；`401/403/404/422/429/500` 语义清晰。

## 7. 非功能需求

### 7.1 性能与扩展

- 列表接口 **默认分页**（如 `limit` 上限 50）。  
- 热点列表/详情：可选 **Redis** 缓存 + **TTL** + **发布时失效**。  
- 数据库只读副本：**首版非必须**；在 PRD 中标注「流量达标再启用」。

### 7.2 安全

- 全站 **HTTPS**。  
- **管理员接口**与**公开接口**路由分离或中间件隔离。  
- **Markdown/HTML** 若允许原始 HTML，需 **消毒（sanitize）** 策略；否则仅允许安全子集。  
- **速率限制**：登录、注册、管理写接口 **更严**。  
- **依赖与镜像**：定期安全更新策略（流程级）。

### 7.3 隐私

- 日志 **脱敏**（密码、token 不落日志）。  
- 若统计访客，遵守告知与最小必要原则（与前台分析工具协调）。

### 7.4 备份与恢复

- 数据库 **每日自动备份** + **保留期**（如 7/30 天）。  
- **恢复 Runbook**：从备份恢复到新实例的步骤文档化。

## 8. 管理后台（浏览器）

- **功能**：登录；文章列表（状态筛选）；编辑器（Markdown 为主推荐）；保存草稿；发布；可选删除/归档。  
- **非功能**：管理端 **独立部署路径**（如 `/admin`）或子域；**不**将管理员入口暴露在公开 API 文档的默认示例中。  

---

## 9. 验收标准（后端首版）

1. 完成 **读者注册/登录** 与 **角色** 区分，读者 **无法** 读写他人草稿或调用管理接口。  
2. 管理员在浏览器中 **新建草稿 → 发布**，匿名用户 **可访问已发布详情**，**不可访问草稿**。  
3. `GET` 列表/详情在压测或脚本下 **p95 延迟**符合第 1.3 节目标量级（可在预发布环境验证）。  
4. 发布后缓存（若启用）**不会长期返回旧正文**（验证失效策略）。  
5. 提供 `/health`、基础监控对接说明、备份与恢复 **Runbook 链接或同仓库文档**。  
6. 与 `PRD.md` 选定的前台衔接模式 **跑通一次端到端**（含 SEO 检查点：详情页可被爬虫抓取或见静态 HTML，视所选模式而定）。

## 10. 风险与开放问题

| 风险 | 缓解 |
|------|------|
| 自建后台研发量 | 优先最小可用：Markdown + 基础字段；富文本后置 |
| XSS / 内容注入 | 严格消毒或禁止原始 HTML |
| 发布与静态站一致性 | Webhook 失败重试与告警 |
| 单点故障 | 托管商多可用区、定期备份 |

**开放问题（实现前确认）**

- 是否开放公开注册，还是 **仅邀请/仅管理员创建读者账号**？  
- 文章正文是否 **仅 Markdown**，是否允许嵌入自定义组件？  
- 前台最终选型 **A/B/C**（见 §3.3）与域名、CDN 策略。

## 11. 附录：名词表

- **BFF**：Backend for Frontend，为特定客户端聚合数据的 API 层。  
- **Webhook**：事件 HTTP 回调，常用于「发布成功 → 触发构建」。  
- **p95 延迟**：95% 的请求不高于该耗时。  
- **RLS**：行级安全（若使用 PostgreSQL 且直连，可加强数据隔离）。  
- **降级**：高负载时关闭非核心能力，保障只读可用。
