# PRD：技术向个人博客（Astro / SSG）

- **文档版本**：v0.2
- **日期**：2026-04-11
- **产品形态**：面向公共互联网读者的站点；**前台以 Astro 交付（SSG 或混合，见下文）**
- **技术选型（已定）**：Astro，默认不引入 React
- **数据源（已定）**：**文章与发布状态以后端 / 数据库为主**；本仓库中的 Markdown/MDX **仅用于占位、迁移或本地开发示例**，不作为线上权威稿源。账户、发布、权限等见 **`docs/PRD-backend.md`**。

---

## 1. 背景与目标

### 1.1 背景

个人技术博客需要长期可读、可被搜索引擎收录、在较高访问量下仍保持较快首屏与稳定体验。内容以中文为主，包含代码与数学公式。**正文与元数据由后端持久化**；前台负责展示、SEO 与性能。

### 1.2 产品目标

- **可读性**：文章排版清晰，代码与公式渲染正确，暗色模式可用（若主题支持）。
- **可发现性**：具备基础 SEO（标题、描述、规范链接、站点地图、RSS）。
- **性能与稳定性**：在静态托管 + CDN（及必要时的 API/CDN）场景下，遵循现代 Web 性能实践，关注 Core Web Vitals。
- **可维护性**：**作者在浏览器中通过管理后台维护内容**；前台构建或拉取流程与 **`PRD-backend.md`** 选定的衔接模式一致，可重复、可自动化。

### 1.3 成功指标（上线后可量化）

| 指标 | 目标（首版建议） |
|------|------------------|
| LCP（移动端） | 力争「良好」区间（以 Search Console / PageSpeed 实测为准） |
| CLS | 尽量避免布局偏移（图片/嵌入物指定尺寸） |
| INP | 控制非必要脚本，第三方脚本按需、延迟加载 |
| 构建 | `build` 可在一台普通开发机上稳定完成；CI 可复现 |

> 注：具体阈值随托管商与网络环境变化，以 **字段数据（真实用户）** 为主，实验室数据为辅。

## 2. 范围

### 2.1 In Scope（首版）

- 首页：文章列表（按发布时间倒序）、必要站点信息入口。
- 文章详情页：标题、日期、标签/分类（若启用）、正文（**中文排版**）、**代码高亮**、**数学公式**。
- 列表与详情 URL 稳定、可分享。
- **RSS/Atom** 订阅。
- **sitemap.xml**、**robots.txt**（与公开环境一致）。
- 基础 **SEO**：每页 `title`/`description`、规范 URL（canonical）、Open Graph 基础字段（可选但推荐）。
- **前端性能策略**（见第 6 节）：图片与字体、缓存依赖静态托管能力、尽量少 JS。
- 部署目标：**静态文件** + **CDN**（具体厂商在实现阶段选定，本 PRD 不绑定）。

### 2.2 Out of Scope（首版不做）

- **评论**、点赞、站内通知（读者账号与登录若启用，**以后端 PRD 为准**）。
- 复杂站内搜索集群（首版可用列表分页；独立搜索服务见路线图）。
- 付费墙、会员体系。
- 多语言（首版仅中文内容；界面文案可中文为主）。

### 2.3 假设

- **权威内容**存于**数据库**（由后端 API 与管理后台写入）；**发布**通过管理后台完成，并可触发 **`PRD-backend.md`** 中的构建或渲染链路。
- Git 仓库主要承载 **Astro 工程、样式与流水线**；文章数据可经 CI 从后端拉取再 `build`，或采用混合渲染，以后端方案为准。
- 访问量增长依赖 **CDN 与缓存**；**后端 API 与数据库**由 `PRD-backend.md` 规划（应用服务器与「仅静态托管」可并存：静态前台 + 独立 API 域名）。

## 4. 信息架构（建议）

- `/` 首页
- `/posts/` 或 `/blog/` 文章列表（与实现一致即可）
- `/posts/[slug]/` 文章详情
- `/tags/[tag]/`（可选）标签列表
- `/rss.xml`（或约定路径）
- `/sitemap-index.xml` + 分页 sitemap（若文章量大）

## 5. 功能需求

### 5.1 内容模型

- 文章字段至少包含：`title`、`date`、`slug`、正文、`draft`（可选）、`tags`（可选）、`description`/`excerpt`（用于列表与 SEO）；**以后端 schema 与 `PRD-backend.md` 为权威**，本节描述前台展示所需视图。
- 正文来源为 **后端存储的 Markdown**（或经消毒的 HTML）；前台渲染时：代码块语言标注正确时由 **Shiki**（或等价）高亮。
- 数学公式：行内与块级公式在页面生成时渲染（如 KaTeX），要求打印与暗色模式下可读。

### 5.2 列表与分页

- 列表页默认按时间倒序；文章量大时支持分页或「加载更多」（首版可用分页以降低单页体积）。

### 5.3 站内导航

- 全局导航：首页、文章列表、RSS、关于页（可选）。
- 文章页：面包屑或返回列表（可选，提升可用性）。

### 5.4 SEO

- 每页独立 `title` 与 `meta description`（文章页用摘要或首段）。
- 文章 URL 人类可读（slug），变更 slug 时评估重定向策略（首版可文档说明「尽量避免改 slug」）。
- 结构化数据：文章类 `Article` / `BlogPosting` JSON-LD（推荐，首版可简化）。

### 5.5 订阅与发现

- 全站 RSS/Atom 输出完整条目或摘要（需在 PRD 或站点说明中固定一种，避免读者困惑）。
- `sitemap.xml` 包含主要页面与文章 URL。

## 6. 非功能需求（高流量向「前端架构」要点）

> 本节对应业界常见技术博客实践：**静态预渲染 + CDN、控制 JS、优化 LCP/CLS、合理缓存**。

### 6.1 渲染与运行时

- **以后端为主**：列表/详情数据来自 **后端 API 或构建前同步的数据**（与 `PRD-backend.md` §3.3 选定模式一致）。
- **SSG / 混合**：可为「构建期拉取已发布文章生成静态 HTML」，或「请求时 SSR/岛屿组件拉取 API」；无论哪种，**读者看到的已发布内容须与数据库一致**。
- **JavaScript**：默认最小化；仅在为明确交互或第三方集成必要时使用 **Astro Islands**，避免整页客户端框架化。
- **无 React**（已定）：不引入 React 运行时。

### 6.2 资源与性能

- **图片**：使用 Astro 生态推荐的图片能力（如 `@astrojs/image` 或官方图片组件策略），输出现代格式（WebP/AVIF，视托管支持），响应式 `srcset`，**显式宽高**以避免 CLS。
- **字体**：优先系统字体栈；若用 Web 字体，需 `font-display` 策略，控制子集与请求量，避免阻塞首屏。
- **第三方脚本**：分析/广告等默认 **延迟加载**，且需可配置开关；首版可仅接入一种轻量统计或暂不接入。

### 6.3 缓存与传输（依赖托管平台配置）

- 静态 HTML 与带 hash 的前端资源：**长期缓存**策略可接受。
- HTML 文档：按平台建议设置较短缓存或按需刷新（避免读者长期看不到更新）。
- 开启 **Brotli/Gzip**（通常由 CDN 提供）。

### 6.4 安全与隐私（前端侧）

- 外链 `rel="noopener noreferrer"`（若在新窗口打开）。
- 内容安全：若前台仍使用 MDX 组件，需限制危险嵌入；**正文来自管理员发布时，仍须按后端 PRD 做存储与展示侧消毒**，读者评论类 UGC 首版不做。

### 6.5 可访问性（基础）

- 语义化标题层级（h1–h3）。
- 图片 `alt` 文本。
- 键盘可聚焦的交互控件（若有）。
- 代码块可横向滚动，不撑破布局。

### 6.6 兼容性

- 支持最新两个大版本的 Chrome / Edge / Firefox / Safari（桌面与移动）。
- 中文显示无乱码，标点与行宽在常见屏幕下可读。

## 7. 技术架构说明（实现约束）

- **框架**：Astro。
- **内容（权威）**：**后端数据库**；实现上可为「构建脚本从 API 拉取已发布文章写入临时目录再 SSG」，或「运行时请求 API」，由 **`PRD-backend.md` 衔接模式**锁定。
- **本仓库 `src/content`**：仅作示例、迁移或离线开发，**非线上一手数据源**（与 v0.2 数据源约定一致）。
- **样式**：与所选主题一致（可 Tailwind 或 Scoped CSS），需满足暗色模式（若提供）。
- **构建**：Node LTS；锁文件提交仓库，保证 CI 可复现。
- **部署产物**：至少包含可托管的 **静态资源**；若存在 SSR/边缘渲染，部署形态在实现方案中说明。

---

## 8. 验收标准（首版）

1. 本地与 CI 执行构建无错误（或与选定混合模式等价的发布流程），产出可部署产物。
2. 在**后端已发布**的文章中随机抽 3 篇，含**代码块**与**公式**，前台渲染正确、无严重样式错位。
3. 首页与文章页在移动与桌面下无显著 CLS（人工检查 + Lighthouse 辅助）。
4. `rss.xml` 与 `sitemap.xml` 可访问且链接有效（数据源与后端一致或经约定生成）。
5. 文章页「查看源代码」可见主要正文文本（利于 SEO 与无 JS 场景；若详情为 CSR 拉取，则须在 PRD 中单独声明并接受 SEO 方案）。
6. **与 `PRD-backend.md` 联调**：管理后台发布后，读者侧在约定时间内可见**同一**已发布内容（取决于 Webhook/缓存策略）。

## 10. 风险与依赖

- **前后端一致性**：发布延迟、缓存旧文、构建失败等依赖 **`PRD-backend.md`** 中的衔接与告警。
- **公式与渲染插件兼容性**：需在实现阶段固定版本组合并做一次回归样本集。
- **托管平台差异**：图片优化 API、Edge 功能因平台而异，需在部署文档中写明。
- **指标达标**：高访问量下瓶颈多在第三方脚本、大图与 **API/数据库**，需持续审计。

## 12. 附录：名词表

- **SSG**：构建时生成静态 HTML，访问时直接由 CDN 提供。
- **CDN**：把静态资源分发到离用户近的节点，降低延迟。
- **Core Web Vitals**：Google 推广的一组体验指标（如 LCP、INP、CLS）。
- **CLS**：布局意外移动，常因图片无尺寸等原因引起。
- **LCP**：视口内最大内容绘制完成时间，常与首屏大图/字体相关。
